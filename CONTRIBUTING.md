# Соглашения для разработки

## Ветки

Имя ветки имеет вид `<тип>/<краткое-описание-в-kebab-case>`.

- Используй только строчные латинские буквы, цифры и дефисы в описании.
- Указывай один тип и одно краткое описание без дополнительных слешей.
- Описание называет изменение, а не задачу в общем виде: `chore/setup-biome`, а не `chore/changes`.

| Тип | Когда использовать | Пример |
| --- | --- | --- |
| `feature` | Новая пользовательская возможность | `feature/cloud-export` |
| `bugfix` | Исправление пользовательской ошибки | `bugfix/calendar-refresh` |
| `chore` | Обслуживание проекта: инструменты, зависимости, конфигурация | `chore/setup-biome` |
| `docs` | Только документация | `docs/contributing-guide` |
| `refactor` | Изменение внутренней структуры без изменения поведения | `refactor/calendar-data-loading` |
| `test` | Тесты или тестовая инфраструктура | `test/sync-queue` |
| `ci` | Автоматизация сборки, проверок или деплоя | `ci/cache-npm` |

`chore` происходит от английского слова *chore*: регулярная хозяйственная или
обслуживающая работа. В разработке это изменения, которые поддерживают проект,
но не добавляют пользовательскую функцию.

## Коммиты

Делай один сфокусированный коммит на одно законченное изменение. Перед коммитом
pre-commit запускает `npm run check`. Каждый коммит, который создаёт агент,
должен иметь сообщение на русском в таком формате: короткое общее название на
первой строке, пустая строка, затем 2-4 коротких пункта с деталями.

Название формулируй как результат в форме краткого страдательного причастия.
Используй прямой порядок слов: сначала причастие, затем объект действия:
`Завершена настройка Biome`, `Уточнён формат сообщений`, `Применена проверка`.
Не используй инфинитивы: `Настроить`, `Уточнить`, `Применить`, и не ставь
причастие после объекта: `Настройка Biome завершена`.

Когда нужно показать пользователю название, описание или текст коммита, выведи
полное сообщение коммита в текстовом кодовом блоке для копирования.

```text
Краткое описание изменения

- Первая важная деталь
- Вторая важная деталь
```

## Pull request

PR направляй в `main`. В описании укажи видимое пользователю изменение, команды
проверки и связанные задачи. Для изменений интерфейса добавь скриншот или короткую
запись. Отдельно отметь изменения схемы Supabase и новые переменные окружения.

## Feature-Sliced Design

Проект использует FSD v2.1. Импорты идут только в нижние слои:
`app` -> `pages` -> `features` -> `shared`. Не создавай слой или slice заранее:
код, используемый одной страницей, остаётся в этой странице. Выноси его ниже
только после подтверждённого повторного использования. Внешние импорты страниц и
features выполняй через их `index.ts`.
